# Journal Finder — 정확도 평가 방법과 기록

> 이 문서는 "검색이 잘못됐을 때 어떻게 고쳐나가는가"에 대한 답입니다.
> 눈으로 보고 "그럴듯하다"고 판단하는 것은 검증이 아닙니다. 정답이 있는 데이터로 채점해야 합니다.

---

## 1. 핵심 원칙

**정답지(gold set)**: 이미 어느 저널에 실렸는지 아는 논문들.
제목·초록만 알고리즘에 넣고, **실제 게재 저널이 추천 목록 몇 위에 나오는지** 측정합니다.

### 지표

| 지표 | 의미 |
|---|---|
| **Recall@1** | 1위 추천이 실제 게재 저널인 비율 |
| **Recall@10** | 상위 10개 안에 실제 저널이 있는 비율 ← **주 지표** |
| **Recall@15** | 상위 15개 안에 있는 비율 (사이트 기본 노출 개수) |
| **MRR** | 정답 순위의 역수 평균. 1위=1.0, 4위=0.25, 목록 밖=0 |

Recall@10을 주 지표로 삼는 이유: 저널 선택은 "정답 하나"가 아니라 **후보 몇 개를 놓고 고르는 일**이기 때문입니다.

### 반드시 지켜야 하는 두 가지

1. **정답 유출(leakage) 방지** — 검색하면 그 논문 자신이 결과에 나와서 자기 저널 점수를 올립니다.
   제목이 일치하는 논문은 후보에서 제거합니다. 이걸 안 하면 성능이 실제보다 훨씬 좋아 보입니다.

2. **튜닝셋 / 검증셋 분리** — 파라미터를 맞춘 데이터로 성능을 보고하면 자기 자신을 채점하는 셈입니다.
   정답지를 절반으로 나눠, 튜닝은 앞쪽에서만 하고 최종 수치는 손대지 않은 뒤쪽에서 확인합니다.

### 표본을 뽑는 방법 (중요)

유명 교수의 논문만 모으면 정답이 최상위 저널에 쏠립니다. 그 편향에 맞춰 튜닝하면
**평범한 논문에도 Nature를 추천하는 지금 문제가 오히려 굳어집니다.**

→ OpenAlex `sample=` 파라미터로 **26개 분야에서 무작위·층화 추출**합니다.
조건: 2023-06 이후, 초록 200자 이상, ISSN 보유, `is_core` 등재 저널.

겉보기에는 그럴듯한 추천이었지만, **정답 저널은 15위 안에 단 한 번도 들지 못했습니다.**

### 원인 분리: retrieval인가 ranking인가

| 논문 | 실제 저널 | 후보 풀에 정답 있음? | 1위 추천 |
|---|---|---|---|

**결론: 9/10에서 정답 저널이 이미 후보 풀에 있었다.**
검색(retrieval)은 정상이고, **순위 매기기(ranking)가 고장난 것**입니다.

### 진단된 원인: 대형 저널 편향

기존 점수 공식은 저널이 받은 유사 논문 점수를 **단순 합산**했습니다.

```
score(저널) = Σ (관련도 × 최신성)
```

Nature Communications는 연간 수천 편을 찍어냅니다. 어떤 주제로 검색하든
"유사 논문"이 여러 편 걸리고, 합산 점수가 커집니다. **논문을 많이 낸다는 이유만으로 1위**가 됩니다.

### 적용한 수정: 규모 정규화

```
score(저널) = Σ (관련도 × 최신성) / (저널 총 논문 수)^α
```

α=0이면 기존 동작(규모 편향), α=1이면 순수 비율. 현재 기본값 **α=0.35**.
이 값은 `eval.html`의 α 스윕으로 정답지에서 최적화해야 하며, **아직 검증 전입니다.**

## 3. 발견한 제약: OpenAlex 예산제

평가 도중 발견했습니다.

- 무료 예산: **계정·IP당 하루 $1**
- 요청 단가: **$0.001** → **하루 약 1,000회**
- 초과 시 HTTP 429, UTC 자정에 초기화

영향:

1. 표본 1편당 1~3회 호출 → **하루 300~600편**이 현실적 한계. 1,000편을 하루에 끝낼 수 없음.
2. **사이트 자체의 버그를 발견**: 429가 뜨면 "유사 논문을 찾지 못했습니다"로 잘못 표시되고 있었음. (수정 완료)
3. 백분위 기능이 검색 1회당 **16회**를 호출하고 있었음 → 동료 저널 분포를 한 번만 받아 로컬 계산하도록 변경, **약 5회**로 축소. (수정 완료)
4. API 키 입력란 추가 — 본인 계정 예산 사용. 키는 브라우저에만 저장.

## 4. 매일 밤 실행에 대하여 (현실적인 기대치)

**자동으로 정확도가 오르지는 않습니다.** 이 시스템에는 학습되는 모델이 없습니다.
매일 돌려서 얻는 것은 다음 세 가지입니다.

1. **정답지 확대** → 측정 오차 감소.
   100편이면 Recall@10의 오차범위가 약 ±10%p, 1,000편이면 약 ±3%p입니다.
   표본이 작으면 개선인지 운인지 구분할 수 없습니다.
2. **회귀 감지** — 알고리즘을 고쳤을 때 *나빠진 것*을 즉시 잡아냅니다. 이게 가장 큰 가치입니다.
3. **실패 사례 축적** — 어떤 분야에서 왜 틀리는지 패턴이 보입니다. 개선 아이디어는 여기서 나옵니다.

정확도를 실제로 올리는 것은 **알고리즘 변경**이고, 그건 사람(또는 저)이 합니다.
벤치마크는 그 변경이 진짜 개선인지 판정하는 심판입니다.

### 예산 안에서 돌리는 법

하루 1,000회 한도 → **분야당 12편 × 26분야 = 312편**이 안전한 1회분입니다.
시드를 날짜로 바꾸면 매일 다른 표본이 뽑히고, 결과를 누적하면 정답지가 커집니다.

## 5. 다음에 시도할 개선안 (우선순위)

1. **α 스윕** — 규모 정규화 지수 최적화. (`eval.html` → α 스윕 버튼, API 호출 0회)
2. **순위 할인** — 관련도 원값 대신 `1/log2(순위+1)` 사용. 상위 결과에 더 큰 가중치.
3. **저널당 기여 상한** — 한 저널이 후보 풀을 독식하지 못하게 상위 k편만 반영.
4. **주제 사전확률** — 저널 J가 주제 T를 얼마나 자주 다루는지(`topic_share`)를 점수에 반영.
   OpenAlex `sources.topic_share`에 이미 있는 값. 대형 저널 편향을 더 정밀하게 교정할 수 있음.
5. **초록 없는 논문 대응** — 제목만 있을 때 키워드가 2~3개로 줄어 후보가 부정확해짐.
6. **리뷰 논문** — 위 표에서 유일하게 후보 풀에도 정답이 없던 사례가 리뷰 논문(Applied Sciences)이었음.
   리뷰는 게재 저널 분포가 원저와 다름. 별도 처리 필요.

## 6. 사용법

`eval.html`을 브라우저로 열고:

1. (선택) OpenAlex API 키 입력 — 공용 예산 소진을 피하려면 권장
2. **▶ 벤치마크 실행** — 정답지 수집 → 후보 풀 캐시 → α=0 vs α=현재값 비교
3. **α 스윕** — 캐시된 풀만 사용, **API 호출 0회**로 α를 여러 값으로 재채점
4. **⬇ 결과 CSV** — 날짜별로 저장해 추이 관리

후보 풀 수집(retrieval)과 점수 계산(ranking)을 분리한 이유가 여기 있습니다.
비싼 건 수집이고, 우리가 고치는 건 점수 계산이므로 **한 번 받아 수십 번 실험**할 수 있습니다.

## 7. 정직한 현재 상태

- ✅ 문제를 정량적으로 확인함 (Recall@15 = 0%)
- ✅ 원인을 retrieval이 아닌 ranking으로 좁힘 (정답의 90%가 후보 풀 안에 존재)
- ✅ 수정안(규모 정규화)을 코드에 반영함
- ⛔ **수정안이 실제로 개선인지는 아직 측정하지 못함** — 평가 도중 일일 예산 소진
- ⏭ UTC 자정 이후 `eval.html`로 α 스윕 → 검증셋에서 확인 → 결과를 이 문서에 추가

수정했다는 사실과 개선됐다는 사실은 다릅니다. 후자는 아직 증명되지 않았습니다.
